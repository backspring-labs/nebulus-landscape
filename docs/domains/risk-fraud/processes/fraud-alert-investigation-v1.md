# Fraud Alert Investigation v1

## 1. Objective

Operationally execute the controlled sequence required to receive, triage, investigate, and disposition fraud alerts generated by detection and monitoring systems, ensuring that true positive fraud is identified and acted upon, false positives are cleared efficiently, and all investigation activity is documented to support downstream SAR referral, customer action, and regulatory examination.

## 2. Scope

This process covers:
- receipt and queue assignment of system-generated fraud alerts
- initial triage and priority scoring of alerts
- case opening and evidence assembly for investigation
- detailed investigation and analysis of flagged activity
- disposition decision rendering (confirm, escalate, dismiss)
- post-disposition actions including customer notification, account restriction, and SAR referral
- case closure and documentation retention

This process does **not** own:
- underlying fraud detection model execution (`cap.fraud_detection_and_scoring` generates the alert; this process consumes it)
- SAR filing execution (covered by `proc.sar_filing_v1`)
- payment blocking or reversal execution (`domain.payments`)
- account restriction enforcement (`domain.accounts`)
- customer communication channel delivery (`domain.channels_experience`)

## 3. Trigger

A fraud detection engine, transaction monitoring system, or manual referral generates a fraud alert that enters the alert management queue.

## 4. Entry Conditions

- Fraud detection and scoring models are operational and producing alerts.
- Alert management platform is available and accepting new alerts.
- Case management system is operational for case creation.
- Investigator workforce capacity is available or queue routing is configured.
- Customer and transaction data sources are accessible for evidence assembly.

## 6. Stage-by-Stage Flow

### Stage: `stage.fraud_alert_investigation.alert_receipt_and_queue_assignment`

**Description:** The fraud detection engine or transaction monitoring system generates an alert based on scoring thresholds, rule triggers, or model output. The alert is ingested by the alert management platform, enriched with basic context, and routed to the appropriate analyst queue based on alert type, severity, and workload distribution.

**Capabilities activated:** `cap.fraud_detection_and_scoring`, `cap.alert_triage_and_disposition`

**Systems involved:** `sys.fraud_detection_engine`, `sys.transaction_monitoring_system`, `sys.alert_management_platform`

**Decision points:**
- Does the alert meet the threshold for queue assignment?
- Which queue or analyst pool should receive the alert based on type and severity?
- Is this alert related to an existing open case or investigation?

**Risks:** `risk.alert_queue_overflow`, `risk.misrouted_alert`, `risk.duplicate_alert_for_same_activity`

**Controls:** `ctrl.alert_priority_scoring_model`, `ctrl.sla_based_queue_management`, `ctrl.duplicate_alert_suppression`

**Evidence generated:** Alert ID, alert timestamp, detection source, initial severity score, queue assignment record, related-case linkage if applicable

**Handoff:** Alert assigned to analyst queue for triage. If related to an existing case, linked and routed accordingly.

### Stage: `stage.fraud_alert_investigation.alert_triage_and_initial_scoring`

**Description:** An analyst reviews the alert in the queue, performs initial assessment of the flagged activity against available context, and assigns an initial triage disposition: escalate to full investigation, dismiss as false positive with documented rationale, or hold for additional information.

**Capabilities activated:** `cap.alert_triage_and_disposition`, `cap.fraud_detection_and_scoring`

**Systems involved:** `sys.alert_management_platform`, `sys.customer_master`, `sys.core_banking_platform`, `sys.transaction_monitoring_system`

**Decision points:**
- Does the flagged activity pattern warrant full investigation?
- Is there sufficient context to dismiss as a known false positive pattern?
- Does the alert require immediate escalation due to severity or dollar amount?
- Should the alert be combined with other recent alerts for the same subject?

**Risks:** `risk.false_positive_fatigue`, `risk.missed_true_positive`, `risk.premature_dismissal`

**Controls:** `ctrl.mandatory_triage_review_fields`, `ctrl.false_positive_dismissal_rationale_requirement`, `ctrl.escalation_threshold_rules`, `ctrl.triage_sla_tracking`

**Evidence generated:** Triage decision record, analyst ID, triage timestamp, initial assessment notes, severity adjustment if applicable, dismiss rationale if applicable

**Handoff:** If triaged for investigation, route to case opening. If dismissed, document and close. If held, schedule follow-up.

### Stage: `stage.fraud_alert_investigation.case_opening_and_evidence_assembly`

**Description:** A formal investigation case is opened in the case management system. The investigator assembles relevant evidence including transaction history, customer profile data, account activity, prior alert history, and external data sources to build the investigation package.

**Capabilities activated:** `cap.case_investigation_management`, `cap.account_and_relationship_surveillance`, `cap.transaction_monitoring`

**Systems involved:** `sys.case_management_system`, `sys.customer_master`, `sys.core_banking_platform`, `sys.transaction_monitoring_system`, `sys.evidence_repository`

**Decision points:**
- What is the scope of evidence required based on alert type and severity?
- Are there related subjects, accounts, or transactions that should be included?
- Does the case require external data enrichment (e.g., device intelligence, IP geolocation)?
- Should interim protective actions (holds, restrictions) be requested while investigation proceeds?

**Risks:** `risk.incomplete_investigation_evidence`, `risk.evidence_assembly_delay`, `risk.scope_creep_without_authorization`

**Controls:** `ctrl.mandatory_investigation_checklist`, `ctrl.evidence_completeness_validation`, `ctrl.interim_action_authorization`

**Evidence generated:** Case ID, case opening record, evidence inventory, subject and account linkages, interim action requests if applicable

**Handoff:** Evidence package assembled; case moves to active investigation and analysis.

### Stage: `stage.fraud_alert_investigation.investigation_and_analysis`

**Description:** The investigator conducts detailed analysis of the assembled evidence, traces transaction flows, evaluates behavioral patterns, contacts other departments or institutions as needed, and develops findings that support a disposition recommendation. This stage may involve multiple review cycles, additional evidence requests, and collaboration with specialists.

**Capabilities activated:** `cap.case_investigation_management`, `cap.transaction_monitoring`, `cap.account_and_relationship_surveillance`, `cap.fraud_rules_and_model_governance`

**Systems involved:** `sys.case_management_system`, `sys.transaction_monitoring_system`, `sys.customer_master`, `sys.core_banking_platform`, `sys.evidence_repository`

**Participating domains:** `domain.customer`, `domain.accounts`, `domain.payments`

**Decision points:**
- Does the evidence support a conclusion of fraud, suspicious activity, or benign behavior?
- Is additional evidence or specialist review required?
- Should the investigation scope expand to include additional subjects or accounts?
- Does the activity meet SAR filing thresholds?
- Are interim customer or account actions required during investigation?

**Risks:** `risk.investigation_bias`, `risk.insufficient_analysis_depth`, `risk.delayed_investigation_completion`, `risk.unauthorized_scope_expansion`

**Controls:** `ctrl.investigation_methodology_standards`, `ctrl.investigation_sla_tracking`, `ctrl.scope_change_authorization`, `ctrl.senior_reviewer_consultation_triggers`

**Evidence generated:** Investigation notes, transaction flow analysis, behavioral pattern documentation, third-party consultation records, finding summary, SAR referral recommendation if applicable

**Handoff:** Investigation findings complete; case moves to disposition and decision.

### Stage: `stage.fraud_alert_investigation.disposition_and_decision`

**Description:** The investigator renders a disposition recommendation based on findings. Depending on case severity and institutional policy, the disposition may require supervisory review and approval before becoming final. Disposition outcomes include: confirmed fraud, suspicious activity requiring SAR referral, false positive/no action required, or escalation to senior investigator or law enforcement.

**Capabilities activated:** `cap.alert_triage_and_disposition`, `cap.case_investigation_management`, `cap.suspicious_activity_reporting`

**Systems involved:** `sys.case_management_system`, `sys.alert_management_platform`, `sys.sar_filing_system`

**Decision points:**
- Does the case meet the standard for confirmed fraud?
- Does the activity meet SAR filing thresholds regardless of confirmed fraud determination?
- Does the disposition require supervisory or committee review?
- Should the case be escalated to law enforcement?
- What customer and account actions should accompany the disposition?

**Risks:** `risk.disposition_without_adequate_review`, `risk.sar_referral_missed`, `risk.inconsistent_disposition_standards`

**Controls:** `ctrl.four_eyes_disposition_review`, `ctrl.sar_referral_trigger_rules`, `ctrl.disposition_consistency_monitoring`, `ctrl.disposition_authority_matrix`

**Evidence generated:** Disposition decision record, reviewer/approver sign-off, SAR referral decision, recommended customer/account actions, disposition rationale narrative

**Handoff:** Disposition finalized; case moves to post-disposition actions.

### Stage: `stage.fraud_alert_investigation.post_disposition_actions`

**Description:** Following disposition, the process triggers downstream actions based on the decision: account restrictions or closures are requested from Accounts, payment blocks or reversals are requested from Payments, customer notifications are initiated through Channels, SAR referral packages are sent to the filing team, and law enforcement referrals are prepared if applicable.

**Capabilities activated:** `cap.case_investigation_management`, `cap.suspicious_activity_reporting`, `cap.alert_triage_and_disposition`

**Systems involved:** `sys.case_management_system`, `sys.core_banking_platform`, `sys.sar_filing_system`, `sys.customer_master`, `sys.evidence_repository`

**Participating domains:** `domain.accounts`, `domain.payments`, `domain.channels_experience`

**Decision points:**
- Which downstream actions are required based on the disposition?
- What is the appropriate timing and sequencing of customer-facing actions?
- Should customer notification be delayed pending law enforcement coordination?
- Are account restrictions temporary or permanent?

**Risks:** `risk.delayed_customer_notification`, `risk.premature_customer_tipping`, `risk.incomplete_downstream_action_execution`, `risk.restriction_scope_mismatch`

**Controls:** `ctrl.anti_tipping_controls`, `ctrl.downstream_action_tracking`, `ctrl.restriction_scope_validation`, `ctrl.law_enforcement_coordination_protocol`

**Evidence generated:** Downstream action requests and confirmations, customer notification records, SAR referral package, law enforcement referral records, restriction implementation confirmations

**Handoff:** All downstream actions initiated and tracked; case moves to closure and documentation.

### Stage: `stage.fraud_alert_investigation.case_closure_and_documentation`

**Description:** The case is closed with complete documentation, all evidence is archived to the evidence repository, quality assurance checks are performed, and the case is marked as available for regulatory examination. Metrics are captured for model performance feedback and operational reporting.

**Capabilities activated:** `cap.case_investigation_management`, `cap.fraud_rules_and_model_governance`

**Systems involved:** `sys.case_management_system`, `sys.evidence_repository`, `sys.fraud_detection_engine`

**Decision points:**
- Are all required documentation elements present and complete?
- Have all downstream actions been confirmed as executed?
- Should the case findings feed back into fraud model tuning?
- Is the case subject to continuing activity monitoring?

**Risks:** `risk.incomplete_case_documentation`, `risk.feedback_loop_degradation`, `risk.continuing_activity_missed`

**Controls:** `ctrl.case_documentation_completeness_gate`, `ctrl.alert_to_case_audit_trail`, `ctrl.model_feedback_loop_process`, `ctrl.continuing_monitoring_schedule`

**Evidence generated:** Case closure record, final evidence archive, QA review outcome, model feedback submission, continuing monitoring flag if applicable

**Handoff:** Case closed. If continuing activity monitoring is required, scheduled review is created. Model feedback is submitted to detection engine governance.

## 7. Decision Points

Key decisions across the process:
- Alert meets queue threshold vs suppressed
- Triage: investigate vs dismiss vs hold
- Evidence scope: standard vs expanded
- Investigation finding: confirmed fraud vs suspicious activity vs benign
- SAR referral required vs not required
- Disposition: approved by investigator vs requires supervisory review
- Post-disposition: restrict/close account vs monitor vs no action
- Case closure: complete vs requires continuing activity review

## 8. Alternate Outcomes

- Alert dismissed as false positive at triage with documented rationale
- Investigation reveals benign activity; case closed with no action
- Suspicious activity identified but below SAR threshold; enhanced monitoring applied
- Confirmed fraud with immediate account restriction and law enforcement referral
- SAR filed with continuing activity review scheduled
- Case escalated to senior investigator or fraud committee
- Multiple alerts consolidated into a single complex investigation

## 9. Risks (process-level)

- Alert volume overwhelming analyst capacity leading to missed true positives
- False positive fatigue degrading investigation quality
- Inconsistent disposition standards across investigators
- Inadequate evidence assembly leading to weak SAR narratives
- Delayed downstream actions allowing continued fraudulent activity
- Model drift causing detection blind spots
- Incomplete documentation failing regulatory examination standards

## 10. Controls (process-level)

- Alert priority scoring and SLA-based queue management
- Mandatory triage review fields and dismissal rationale requirements
- Investigation methodology standards and mandatory checklists
- Four-eyes disposition review for cases above threshold
- SAR referral trigger rules and filing deadline tracking
- Anti-tipping controls for customer-facing actions
- Case documentation completeness gate before closure
- Alert-to-case audit trail integrity

## 11. Evidence / Audit Considerations

This process generates a high-density audit trail:
- alert generation and scoring records
- queue assignment and SLA compliance records
- triage decision and rationale documentation
- case opening and evidence inventory
- investigation notes and analysis artifacts
- disposition decision with reviewer sign-off
- downstream action requests and confirmations
- SAR referral packages and filing confirmations
- case closure and QA review records
- model feedback submissions

## 15. Provenance

This process is the primary operational spine for the fraud investigation function within `domain.risk_fraud`. It connects the output of automated detection systems to human investigative judgment and downstream enforcement actions. The process is examination-critical because examiners evaluate alert management, investigation quality, disposition consistency, and SAR referral timeliness as core BSA/AML program effectiveness indicators.
